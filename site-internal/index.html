The idea

You run five coding agents in five terminals. Rubberduck is the one window that tells you what they're all doing — which is busy, which is idle, which one needs you right now — launches them into isolated git worktrees, lets you fork any session into a tree of parallel attempts, and keeps a durable history with an intention → outcome summary per session.

It started as an extraction of uv-suite's "Watchtower" (a passive observer) and grew into an active orchestrator. The observer is now a subset of the operator.

The wedge (why this isn't just another worktree launcher)

Conductor, Superset, ccmanager, Claude Squad, crystal, emdash — they all show you live parallel agents well. Almost none keep durable session history with intention→outcome summaries, and none treat the fork tree (which session descended from which) as a first-class artifact. That history + lineage layer is the differentiator.
Note: superset.sh (~9.2k★, PH #1 Feb 2026) is the closest commercial analog.

Architecture

React dashboard (web/) ──SSE──▶ asyncio HTTP server (src/rubberduck/)
                                  ├─ EventBus: in-mem ring (live) ─▶ SSE
                                  │                              └─▶ HistoryStore (SQLite, durable)
                                  ├─ Orchestrator ─▶ SessionSupervisor (one PTY per agent)
                                  ├─ WorktreeManager (git worktree per session)
                                  ├─ Summarizer (CLI / HTTP / none)
                                  └─ AgentRuntime adapter: generic | claude-code | codex

ingest: Claude Code hooks ──▶ rubberduck-hook.sh ──POST /events──▶ server
storage: ~/.rubberduck/{db.sqlite, worktrees/, snapshots/}

Key decisions (and why)

Python, not Node (the original brief's stack)
Gained PTY/subprocess control, stdlib SQLite, the language to win in. Lost the "zero-dep" + "single-file dashboard" badges — kept the spirit (thin deps, server is stdlib-only).
One state machine, not two
The supervisor maps detected PTY state onto the existing event vocabulary (busy→PreToolUse, idle→Stop, waiting→Notification). So derive_state, the dashboard, and SQLite all just work for both spawned and externally-reported agents. No parallel code path.
Agnostic by default, Claude-specific only where it earns it
Generic + codex paths stay runtime-neutral. Only the claude-code adapter special-cases Claude (hook events + JSONL transcript for richer state/summaries) — documented as a conscious choice.
SQLite, but no message queue / microservices / plugin registry / auth
The orchestration complexity (PTYs, worktrees, durable history) is earned. Distributed-systems machinery is not. One process + SQLite, single dev, localhost — until a concrete trigger says otherwise.
Hooks over terminal-attach (for "watch what I run")
rubberduck install-hooks --global wires Claude Code so every session auto-streams in — no curl, no per-repo setup. This was the missing "ignition": the engine existed, this is what makes it usable.
Slop guardrails enforced in CI
A grep+AST slop_check.py fails the build on vague-adjective docs and existence-only tests. It got smarter mid-build (AST-aware) rather than being silenced when it flagged legitimate type-narrowing guards.

Naming notes

"Rubber duck" is heavily contested in AI dev tooling — GitHub Copilot shipped a "Rubber Duck" feature (Apr 2026), plus rubberduck.com, the VS Code extension, Rubber Ducky Labs (YC). Every short rubberduck.* domain is taken. Decision: keep the name, took rubberduckhq.com. Alternatives explored if it ever needs a pivot: DuckFleet, DuckTower, Covey, DuckRaft (a "raft" is the real collective noun for ducks on water = a fleet).
Open risk accepted: SEO collision + potential trademark friction with Microsoft.

How I run it

rubberduck install-hooks --global   # wire Claude Code once
rubberduck serve                    # start the server
# then just use Claude normally — sessions appear in the dashboard
cd web && npm run dev               # dashboard at localhost:5173
